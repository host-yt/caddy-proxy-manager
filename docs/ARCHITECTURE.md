# Architecture

Hostyt Proxy Gateway is a self-hosted control plane for a fleet of Caddy
reverse-proxy nodes. A single Go binary (~21 MB) manages plans, services,
and domains; translates them into Caddy JSON configs; and pushes those
configs over WireGuard to every node in the fleet.

---

## 1. Overview

```
Browser / API client
        |
        | HTTPS (public)
        v
  [ Caddy node(s) ]   ──── WireGuard mesh ────  [ Manager node ]
   port 80/443                                    port 8080 (internal)
   Caddy Admin :2019                              ┌──────────────────────────┐
   (docker-only)                                  │   hpg-server binary      │
        |                                         │  ┌────────┐ ┌─────────┐ │
        | /internal/ask                           │  │ chi    │ │caddyapi │ │
        └────────────────────────────────────────►│  │ router │ │ client  │ │
                                                  │  └────────┘ └─────────┘ │
                                                  │  ┌────────┐ ┌─────────┐ │
                                                  │  │domain/ │ │  auth   │ │
                                                  │  │services│ │sessions │ │
                                                  │  └────────┘ └─────────┘ │
                                                  └──────┬───────────────────┘
                                                         │
                                               ┌─────────┴──────────┐
                                               │                    │
                                          [ MariaDB ]           [ Redis ]
                                          (source of truth)    (sessions,
                                                                rate limits,
                                                                cache)
```

**Caddy nodes** serve live traffic; their configs are generated by the panel
and delivered over the Admin API (`:2019`). Caddy nodes never connect to
MariaDB directly. The diagram draws the manager and a Caddy node as
separate boxes because that's the general (multi-node) case; the default
single-host compose colocates them - the manager's own bundled Caddy is
node-1 and carries customer traffic too. See `docs/MULTI_NODE.md` § 11.

**MariaDB** is the sole source of truth. Every Caddy resync is a DB read +
Admin API write, never the reverse.

**Redis** backs sessions, CSRF tokens, brute-force counters, rate-limit
sliding windows, and the `/internal/ask` verdict cache.

## 2. Component breakdown

### `cmd/server/`
Thin entrypoint. Loads config, wires all services, starts the HTTP listener
and background loops (leader election, metrics poll, drift reconciler,
backup scheduler). Handles graceful shutdown with a background context
(`bgCtx`) that outlives the HTTP server so in-flight Caddy pushes can drain.

### `cmd/node-agent/`
Lightweight agent binary deployed on remote Caddy nodes. Pulls WireGuard
peer lists from the manager and writes the local `wg0.conf`.

### `cmd/rotate-secret/`
CLI tool for rotating `APP_SECRET` - re-encrypts all AES-256-GCM blobs in
the DB under the new key.

### `cmd/restore/`
CLI tool to restore a DB backup from a configured destination (S3/SFTP/FTP).

### `cmd/sso-jump-gen/`
CLI tool for generating SSO jump tokens for testing the JWT-signed jump URL
mechanism.

### `internal/config/`
Loads the full runtime config from environment variables into a typed
`Config` struct. No file parsing in production; `.env` is dev-only.
Validates required fields (e.g. `APP_SECRET` ≥ 32 chars after install,
`APP_URL`, DB credentials). Exposes module-availability gates
(`CACHE_HANDLER_AVAILABLE`, `LAYER4_AVAILABLE`, `WAF_MODULE_AVAILABLE`,
etc.) that guard non-stock Caddy module emission in the config builder.

### `internal/installstate/`
Manages `data/install_state.json` - the wizard's step-by-step progress file.
DB credentials stored here are encrypted (AES-256-GCM, key derived from
`APP_SECRET` via HKDF-SHA256). When `Installed == true` the wizard routes
return 404.

### `internal/store/`
Opens the MariaDB connection pool (`go-sql-driver/mysql`) and runs goose
migrations at boot. Migrations are embedded in the binary (`embed.go`) so
the binary is self-contained. The pool is lazy: a nil-pool is normal pre-
install.

### `internal/domain/`
Business logic, split by aggregate:

| Sub-package | Responsibility |
|-------------|---------------|
| `routes`    | Route lifecycle: DNS pre-check, node placement, Caddy push (incremental or full `/load`), SSL status tracking, alias expansion, wildcard DNS-01 policy assembly |
| `nodes`     | Caddy node fleet management: registration, health polling, placement heuristic (lowest `current_routes / max_routes` within a group), drain/decommission |
| `plans`     | Plan definitions (`restricted` vs `npm`), feature limit enforcement (max domains, ports, rate-limit RPM) |
| `services`  | Service records linking a client to a plan and a node group; `backend_ip` is never exposed to client users |
| `users`     | User identity; wrapped by the `auth` package for password ops |
| `wgpeer`    | Customer WireGuard peer lifecycle (key negotiation, IP allocation, health handshake reporting) |

### `internal/caddyapi/`
Thin client for Caddy's JSON Admin API (`:2019`).

- `client.go` - HTTP verbs: `Load` (POST /load), `AddRoute`, `ReplaceRoute`
  (PATCH /id/\<tag\>), `DeleteRoute` (DELETE /id/\<tag\>), `GetRaw`,
  `PurgeCache`. Response bodies are capped at 32 MiB to prevent a
  misbehaving node from exhausting panel memory.
- `config.go` - `BuildNodeConfig`: assembles the full Caddy JSON config from
  a `[]Route` + `NodeSettings`. Produces the `apps.http.servers.srv0`,
  `apps.tls.automation`, optional `apps.cache`, and optional `apps.layer4`
  blocks. Wildcard DNS-01 policies are prepended before the on-demand
  catch-all (first-match order).
- `route.go` - `BuildRoute`: per-route handler chain assembly. Order:
  WAF → inbound bearer gate (External routes) → Souin cache → gzip/zstd
  encode → custom handlers → rate_limit → WebSocket gate → SSO
  forward-auth → basic auth → primary (reverse_proxy / static_response /
  redirect) → ACL subroute wrapper → force-HTTPS subroute wrapper.
- `stream.go` - `buildLayer4App`: TCP/UDP stream routes for the
  `apps.layer4` block (requires `caddy-l4` module).
- `dnsproviders.go` - Registry of DNS provider slugs and their caddy-dns
  module names/field schemas (Cloudflare, Route53, Gandi, etc.).
- `errorpage.go` - Brand-aware HTML for `handle_errors` (503 maintenance,
  404, 502/504).

### `internal/httpserver/`
Chi router wiring (`server.go`) and all handlers.

**Middleware stack** (applied globally):
`RequestID` → `TraceID` → `TrustedRealIP` → `CloudflareIP` → `Recoverer`
→ `SecurityHeaders` → `Metrics` → request logger → 30 s timeout →
`installRedirect` → `LoadSession` → `VerifyCSRF` → `UnauthPostLimit`

Route groups:
- `/install/*` - wizard, gated by `InstallGuard`
- `/auth/*` - login, TOTP/SMS/email OTP, passkey, OIDC, password reset
- `/admin/*` - requires role `super_admin` or `admin`; optional
  `RequireAdmin2FA` gate
- `/app/*` - client portal, requires role `client`
- `/api/v1/*` - REST API, bearer token auth via `APIKeyAuth` middleware
- `/internal/ask` - Caddy on-demand TLS permission endpoint
- `/api/wg/*`, `/api/node/*` - WireGuard bootstrap and node-agent endpoints

**Handler groups:**

| Handler file | Surface |
|---|---|
| `wizard.go` | Install wizard steps |
| `auth.go` | Login, logout, password reset, impersonation |
| `admin_hosts.go` | Flat host list (NPM-style), bulk ops |
| `admin_nodes.go` (in `admin.go`) | Node fleet, join token, resync, drain |
| `admin_stats.go` | KPI dashboard, Prometheus-backed charts |
| `admin_clients.go` | Client CRUD, impersonation |
| `admin_tunnels.go` | Customer WG tunnel management |
| `client.go` | Client portal: routes, services |
| `api_v1.go` | REST API v1 (services, routes, nodes) |
| `caddy_ask.go` | `/internal/ask` TLS permission gate |
| `nodejoin.go` | Join token redemption |
| `wg_bootstrap.go` | Customer WG config + installer script |

### `internal/auth/`
- `argon2.go` - Argon2id password hashing (PHC format, 64 MiB / 3 passes /
  2 threads). TOTP secrets and API key hashes use the same verify path.
- `session.go` - Redis-backed sessions keyed `hpg:sess:<id>`. Random 24-byte
  ID, 12-hour TTL, rotated on login. Carries `ImpersonatorUserID` during
  admin impersonation.
- `apikey.go` - API key issue (Argon2id hash) + fast HMAC pre-screen derived
  from `APP_SECRET`.
- `totp.go`, `emailotp.go`, `smsotp.go` - TOTP (HOTP/TOTP RFC 6238),
  email OTP, SMS OTP second-factor flows.
- `webauthn.go` - WebAuthn/passkey registration and authentication.
- `password_reset.go` - Time-limited reset tokens stored as sha256 hashes.

### `internal/httpserver/middleware/`
- `csrf.go` - Synchronizer token pattern. Skips safe methods, install
  routes, and `/api/*`. Uses `crypto/subtle.ConstantTimeCompare`.
- `trusted_realip.go` - Honors `X-Forwarded-For`/`X-Real-IP` only from
  configured `APP_TRUSTED_PROXIES` CIDRs.
- `ratelimit.go` - Redis sliding-window rate limiter. Instances: login
  brute-force, `/internal/ask`, unauthenticated POST global cap.
- `api_key.go` - Verifies bearer token in `Authorization` header; sets
  session-like context.
- `require2fa.go` - Blocks `/admin/*` for admins without any enrolled 2FA
  when the `REQUIRE_ADMIN_2FA` env or the DB toggle is on. Grace-period
  aware.
- `security_headers.go` - CSP, HSTS, X-Frame-Options, etc.
- `cf_ip.go` - Optionally trusts `CF-Connecting-IP` (toggled per DB
  setting, not just env).

### `internal/wireguard/`
Manages the control-plane WireGuard identity (manager mesh key) and writes
peer config files that the WG sidecar container picks up via `wg syncconf`.
Key generation is pure Go (`golang.org/x/crypto/curve25519`), no `wg`
shell calls. AES-256-GCM encrypts the private key at rest in the `settings`
table.

### `internal/nodejoin/`
Mints one-time join tokens (30-minute TTL, SHA-256 stored, prefix-indexed
for lookup). On redemption: creates the `caddy_nodes` row, allocates a WG
IP, registers the node's public key, writes the WG peer config, and kicks
an initial Caddy resync.

### `internal/metrics/`
Scrapes each node's Caddy `/metrics` Prometheus endpoint at a configurable
interval (leader-gated). Accumulates delta counters (`caddy_http_requests_total`
per node/host) into the `traffic_stats` table for the admin dashboard charts.

### `internal/leader/`
Redis `SET NX PX` lock (`hpg:leader`) for background worker singletons.
Prevents duplicate health probes, metrics polls, and drift reconcilers in
hot-standby deployments. Single-replica installs hold leadership trivially.

### `internal/installstate/`
State file at `data/install_state.json`. Each wizard step (DB, admin,
app URL, SMTP, Caddy node) is persisted before proceeding so the wizard
is resumable after a crash. DB credentials are AES-GCM encrypted.

### `internal/audit/`
Writes every state-changing action to `audit_log`. Attributes impersonated
actions to the admin user (`ImpersonatorUserID`) with the client ID in
`meta`. Forwards each event to a SIEM webhook (`AUDIT_SIEM_WEBHOOK`) via
a safe HTTP client that blocks RFC-1918/loopback targets.

### `internal/backup/`
Scheduled database backups. Supports S3 (minio-go), SFTP (pkg/sftp), and
FTP (jlaffaye/ftp) destinations. Backup content is a MariaDB dump; files
are AES-256-GCM encrypted before upload. Scheduler is leader-gated.

### `internal/oidc/`
Wraps `coreos/go-oidc/v3` for OIDC login (Authentik, Microsoft, generic
provider). Provider config (issuer, client ID, secret) is stored in the
`settings` table so it can be changed without a restart.

### `internal/dns/`
Pre-flight DNS resolver used by the "Add host" form and the `/internal/ask`
handler warm path. Resolves A/AAAA records and compares against the target
node's public IP. Informational; Caddy issues the certificate regardless.

### `internal/obs/`
Prometheus metrics (`obs.Metrics`) and a structured health handler
(`obs.Health`). Exposes `/metrics` (CIDR-gated) and `/healthz` / `/readyz`.

### `internal/queue/`
Redis-backed async job queue (asynq). Defers DNS verify, SSL retry,
route propagation, and email sends off the request path.

### `internal/i18n/`
Cookie-based language selection; templates carry translated strings.

### `internal/cloudflare/`, `internal/captcha/`
Cloudflare API integration (zone/DNS ops) and Turnstile CAPTCHA verification.
Both read config from the `settings` table at runtime.

### `internal/mail/`, `internal/sms/`, `internal/notify/`, `internal/webhook/`
SMTP email (go-mail), SMS OTP dispatch, unified notification dispatcher,
and admin-configured outbound webhooks (fired on audit events).

### `internal/integration/`
FOSSBilling / Hostyt billing platform adapter used by the REST API v1.

### `internal/validate/`
Input validators shared across handler layers (domain format, port ranges,
CIDR checks, etc.).

### `internal/adminscope/`, `internal/reseller/`
`adminscope` resolves the effective admin scope per request: unrestricted
(super-admin), reseller-scoped, client-scoped, or fail-closed-denied when the
admin's reseller is suspended. `reseller` is the reseller domain package
(resellers, branding, plan/client ownership, and reseller-package CRUD in
`plans.go`). The in-repo `terraform-provider-hpg/` is a nested Go module that
talks to `/api/v1` to manage nodes, pools, plans, clients, services, and
routes as code.

Admin tiers, most to least privileged:

1. **`super_admin`** - user id `0`-equivalent unrestricted path; full platform
   access, unfiltered by `adminscope`.
2. **Platform admin** - `role = 'admin'`/`'api'` with `users.reseller_id`
   NULL and `users.is_restricted = 0`; full platform access except
   `super_admin`-only settings.
3. **Reseller-admin** - `role = 'reseller'` (or legacy `role = 'admin'` with
   `users.reseller_id` set); scope is exactly the clients owned by that
   reseller (`clients.reseller_id`), resolved fresh on every request - never
   cached, never widened by a stray `admin_client_scope` row. A suspended
   reseller (`resellers.status = 'suspended'`) fails its admins closed to
   nothing, not through to unrestricted.
4. **Client-scoped admin** - `role = 'admin'` with `users.is_restricted = 1`;
   scope is the client ids listed in `admin_client_scope`. Restriction is an
   explicit flag (not inferred from row count), so a scoped admin with zero
   assigned clients correctly sees nothing instead of silently escalating to
   full access.

### `internal/quota/`
Enforces a reseller's aggregate package limits (`reseller_plans`) at the three
create surfaces that grow reseller usage: clients, services, and routes. A
business limit, not a security boundary - tenant isolation is `adminscope`'s
job. `Service.CanCreateClient`/`CanCreateService`/`CanCreateRoute` return
typed errors (`ErrClientQuota`, `ErrServiceQuota`, `ErrDomainQuota`) that
handlers map to `403`. Platform-direct resources (`reseller_id` NULL) and
resellers with no package subscribed are never limited (the "Unlimited"
package backfill keeps existing installs unaffected). Overselling
(`resellers.overselling_allowed`) switches domain enforcement from an
allocation check at service-create time (sum of attached plans' `max_domains`
against the package total) to a real route count check at route-create time.
`Service.UsageFor` powers the reseller dashboard/API usage summary.

## 3. Data model

Core entity relationships (abridged to the most significant columns):

```
settings (key, value, is_encrypted)

resellers (id, name, slug, status active|suspended, brand_name, logo_url,
           support_email, primary_color, reseller_plan_id*, owner_user_id,
           overselling_allowed, can_create_plans)   (* NULL = no package = unlimited)
  └── reseller_plans (id, name, max_clients, max_domains_total,
                      max_services_total, rate_limit_rpm_cap)   -- 0 = unlimited
        └── reseller_plan_node_groups (reseller_plan_id, node_group_id)
        └── reseller_plan_features (reseller_plan_id, feature)

users (id, email, password_hash, role super_admin|admin|support|client|api|reseller,
       totp_enabled, oidc_subject, reseller_id*, is_restricted)   (* NULL = platform-direct)
  └── recovery_codes (user_id, code_hash, used_at)
  └── api_keys (user_id, key_prefix, key_hash, scopes)
  └── clients (user_id, display_name, external_ref, reseller_id*)
        └── services (client_id, backend_ip*, allowed_port_start*, allowed_port_end*,
                      plan_id, node_group_id, status)   (* admin-only fields)
              └── routes (service_id, caddy_node_id, domain, path_prefix,
                          upstream_port, status, lb_policy, ...)
                    └── route_upstreams (route_id, host, port, weight)
        └── customer_wg_peer (client_id, node_id, pubkey, assigned_ip, status)
              └── customer_wg_bootstrap (peer_id, token, consumed_at, expires_at)

node_groups (id, name, mode)           -- single / active_active / failover
  └── caddy_nodes (node_group_id, name, api_url, wg_ip, wg_public_key,
                   tunnel_enabled, tunnel_transport, tunnel_wstunnel_port, ...)
  └── plans (node_group_id, name, kind, max_domains, max_ports, ssl_enabled,
             reseller_id*, ...)   (* NULL = global plan)

stream_routes (service_id, caddy_node_id, protocol, listen_port, upstream_port)
node_join_tokens (token_hash, node_group_id, expires_at, used_at, used_node_id)
audit_log (user_id, actor_type, action, entity, entity_id, ip, meta)
traffic_stats (node_id, domain, period_start, bytes_in, bytes_out, request_count)
alert_log (...)
```

**Plan kinds** (`plans.kind`):
- `restricted` (default) - admin pins `backend_ip` and `allowed_port_start/end`
  on the service; the client only picks a domain and a port within that range.
- `npm` - client may edit the service's `backend_ip` and port range
  themselves (reseller / own-account use).

**Route status lifecycle:**
`pending_dns` → `dns_ok` → `pending_ssl` → `active` / `failed` / `disabled`

DNS pre-check and Caddy push can advance the status; manual retry resets
it to `dns_ok` for a fresh push.

## 4. Request flow (browser portal)

```
Browser
  │ POST /admin/hosts/new  (form, cookie session)
  │
  ▼
chi router  (httpserver/server.go)
  │ global middleware: RealIP, SecurityHeaders, LoadSession, VerifyCSRF,
  │                    UnauthPostLimit
  │ group middleware: RequireRole("admin")
  │
  ▼
handlers.AdminHandlers.HostsCreate  (handlers/admin_hosts.go)
  │ - parse + validate form (internal/validate)
  │ - DNS pre-check (internal/dns)
  │ - find or create client/service rows (domain/services.Service)
  │ - write audit_log (internal/audit)
  │
  ▼
domain/routes.Service.Create  (domain/routes/service.go)
  │ - validate port within plan range
  │ - pick caddy_node (lowest utilisation in node group)
  │ - INSERT routes row (status=pending_dns)
  │ - fire background goroutine: DNS verify → Caddy push
  │
  ▼  (background)
domain/routes.Service.pushToNode
  │ - SELECT all active routes for that caddy_node
  │ - caddyapi.BuildNodeConfig(routes, NodeSettings)
  │ - caddyapi.Client.Load(ctx, config)  →  POST http://<wg_ip>:2019/load
  │   OR incremental: Client.ReplaceRoute / Client.AddRoute
  │
  ▼
Caddy node (Admin API :2019)
  - atomically swaps running config
  - starts On-Demand TLS issuance when first request arrives for the domain
```

For the REST API v1 the path is the same from `domain/routes.Service` onward;
the handler is `handlers.APIHandlers` and the auth is `mw.APIKeyAuth` instead
of cookie+CSRF.

## 5. Caddy integration

### Config delivery
The panel never modifies Caddy's `Caddyfile`. It talks exclusively to
Caddy's JSON Admin API (`CADDY_ADMIN_URL`, default `http://caddy:2019`).

- **Full resync**: `POST /load` with the complete config object generated by
  `caddyapi.BuildNodeConfig`. Used when a node is first registered, when its
  WG key rotates, or when an admin triggers "Resync node".
- **Incremental push** (default, `INCREMENTAL_PATCH=1`): single-route
  changes use `PATCH /id/route_<id>` or `POST /config/.../routes` + optionally
  `DELETE /id/route_<id>`. Falls back to a full `/load` on any error.

### On-Demand TLS / `/internal/ask`
Caddy's `tls.automation.on_demand.permission` points to
`http://app:8080/internal/ask?domain=<domain>`.

`AskHandler`:
1. Validates domain shape (max 253 chars, contains dot).
2. Rate-limits per source IP via Redis (`hpg:ask:rl:<ip>`, configurable,
   default 120/min).
3. Checks a Redis decision cache (`hpg:ask:dec:<domain>`): allow cached 120 s,
   deny cached 15 s.
4. On cache miss: `SELECT COUNT(*) FROM routes WHERE domain=? AND status IN
   ('dns_ok','active','pending_ssl') AND ssl_enabled=1` (uses
   `idx_route_domain`). Also allows the node's wstunnel hostname when
   WSS transport is configured.
5. Returns 200 (allow) or 403 (deny). Caddy issues the Let's Encrypt cert
   only on 200.

### Wildcard DNS-01
When `DNS01_AVAILABLE=1` and a DNS provider is configured for a zone,
`BuildNodeConfig` prepends per-zone automation policies with a `dns`
challenge issuer (caddy-dns module). These policies carry explicit `subjects`
(`["*.zone", "zone"]`) and appear before the on-demand catch-all
(`on_demand: true`) so Caddy evaluates them first-match.

### Module feature gates
Six boolean flags in `config.CaddyConfig` (and matching `NodeSettings`
fields) prevent the config builder from emitting handler blocks that stock
`caddy:2.8` does not know. Emitting an unknown handler causes Caddy to
reject the entire `/load`, taking the node offline. Flags default off; the
operator enables them only after rebuilding every node's Caddy image.

| Env | Gates |
|-----|-------|
| `CACHE_HANDLER_AVAILABLE` | `apps.cache` + per-route `cache` handler (Souin) |
| `LAYER4_AVAILABLE` | `apps.layer4` (TCP/UDP stream routes) |
| `WEIGHTED_LB_AVAILABLE` | `weighted_round_robin` LB policy |
| `RATE_LIMIT_AVAILABLE` | per-route `rate_limit` handler (caddy-ratelimit) |
| `WAF_MODULE_AVAILABLE` | per-route `waf` handler (coraza-caddy) |
| `DNS01_AVAILABLE` | wildcard DNS-01 automation policies (caddy-dns) |

## 6. WireGuard mesh

### Control-plane mesh (manager ↔ Caddy nodes)
The manager generates a Curve25519 keypair (pure Go, no `wg` binary) stored
AES-GCM encrypted in the `settings` table. Each remote Caddy node gets a
unique WG IP from a configurable subnet; `caddy_nodes.wg_ip` and
`caddy_nodes.wg_public_key` persist the allocation.

The WG sidecar container (`deploy/wireguard/`) runs a watch loop that calls
`wg syncconf` when the peer list changes. The panel writes
`/etc/wireguard/wg0.conf` (via the `wireguard.Service.WriteConfig` callback
in `nodejoin`), the sidecar picks it up without restarting the interface.

**Node-join flow:**
1. Admin generates a one-time token (`nodejoin.Service.Mint`), 30-min TTL,
   stored as SHA-256 in `node_join_tokens`.
2. VPS runs `curl https://panel/install/node.sh | sudo bash`. The script
   installs WireGuard, Docker, Caddy, then calls
   `POST /api/v1/nodes/join` with the plaintext token.
3. `nodejoin.Service.Redeem`: verifies hash, inserts `caddy_nodes`, allocates
   WG IP, registers public key, writes peer config, marks token used.
4. Panel kicks a full Caddy resync on the new node.

### Customer WireGuard tunnels (optional)
Per-node per-client WG peers for backends that are behind NAT. The node runs
a second WG interface (`tunnel_listen_port`). A client generates a keypair in
the browser, submits the public key; the panel assigns an IP from
`tunnel_subnet`, writes the peer to `customer_wg_peer`, and persists the
server-side private key encrypted in `customer_wg_peer.server_privkey_e2`
(AES-256-GCM). A route's `via_wg_peer_id` ties it to a peer so
`domain/routes` assembles the correct backend IP at push time.

### WireGuard-over-WebSocket (wstunnel, migration 51)
When UDP port 51820 is firewalled on the remote VPS, the tunnel can use
WebSocket transport (`tunnel_transport = 'wss'`). The node runs wstunnel
server on `tunnel_wstunnel_port`; Caddy's `reverse_proxy` routes
`<node_hostname>/wg-tunnel*` to it. The panel adds this Caddy route
automatically when `WstunnelRoute` is populated in `NodeSettings`.
The node reports its wstunnel health back to the panel via the node-agent;
the panel gates the WSS config on `tunnel_wstunnel_healthy`.

## 7. Security layers

| Layer | Mechanism |
|-------|-----------|
| Password storage | Argon2id, 64 MiB / 3 iterations / 2 threads, PHC format |
| Sessions | Random 24-byte ID, Redis-backed, 12-hour TTL, rotated on login |
| CSRF | Per-session synchronizer token; `crypto/subtle.ConstantTimeCompare`; skips `/api/*` and safe methods |
| API key auth | Argon2id hash; HMAC pre-screen from `APP_SECRET` for fast rejection |
| Rate limiting | Redis sliding window: login per (email, IP), `/internal/ask` per IP, global unauthenticated POST cap |
| Brute-force lockout | Incremental Redis counter on failed logins; locks the account after threshold |
| Secrets at rest | AES-256-GCM (NONCE prepended, HKDF-SHA256 key from `APP_SECRET`) for DB password in wizard state, WG private keys, DNS provider credentials, customer tunnel server keys, external route proxy secrets |
| TLS | Caddy on-demand TLS; Let's Encrypt (ACME HTTP-01 default, DNS-01 for wildcards); staging flag for testing |
| CAPTCHA | Cloudflare Turnstile, optional, DB-configurable without restart |
| 2FA | TOTP (pquerna/otp), SMS OTP, email OTP, recovery codes (stored as argon2id hashes), WebAuthn/passkeys |
| OIDC | coreos/go-oidc/v3; provider config in `settings` table |
| IP trust | `TrustedRealIP` middleware honors `X-Forwarded-For` only from `APP_TRUSTED_PROXIES` CIDRs; `CloudflareIP` middleware is separately gated |
| Admin 2FA enforcement | `REQUIRE_ADMIN_2FA` env or live DB toggle gates `/admin/*` until the operator has at least one 2FA method enrolled; grace-period window avoids instant lockout |
| SIEM forwarding | Every `audit_log` event is POSTed to `AUDIT_SIEM_WEBHOOK` via a safe HTTP client that blocks RFC-1918 and loopback targets |
| External upstream allowlist | `EXTERNAL_UPSTREAM_ALLOWLIST` controls which FQDNs may be used as an external-HTTPS upstream; the primary open-relay defense |
| Security headers | `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options` applied globally |
| `/metrics` CIDR guard | `APP_METRICS_ALLOW` restricts the Prometheus endpoint to operator-configured CIDRs |
| Caddy Admin API isolation | `:2019` is bound to `0.0.0.0` inside the container but `docker-compose.yml` deliberately never publishes port 2019; only WG-internal traffic reaches it |

## 8. Key invariants

**`backend_ip` is admin-only.**
`services.backend_ip`, `services.allowed_port_start`, and
`services.allowed_port_end` are set by admins at service-create time.
Client users (role `client`) cannot modify these fields from `/app/services`
unless the service's plan has `kind = 'npm'`. The domain service layer
enforces this before any DB write.

**Caddy Admin API `:2019` is never publicly reachable.**
The Docker Compose file does not publish port 2019. Caddy nodes are
reachable only from the manager via the WireGuard mesh IP. The admin
config comment in `BuildNodeConfig` explicitly documents this invariant:
`"listen": "0.0.0.0:2019"` inside the container maps to the docker bridge,
not the host network.

**MariaDB is the single source of truth.**
The panel never reads running Caddy config to infer state. All state
transitions (DNS check, SSL issuance timestamps, route status) are written
to MariaDB first; Caddy is then driven from that DB state. A full resync
rebuilds the entire node config from DB rows; partial sync patching a
single `@id` is a performance optimization, not a separate state source.

**Module gates: never emit before fleet is ready.**
Feature-gated handler blocks (cache, layer4, rate_limit, WAF, DNS-01) are
never emitted until the corresponding `*_AVAILABLE` env is set. Flipping a
gate before all nodes run the custom Caddy image causes those nodes to
reject `/load` and go offline. The invariant is: gate on only after every
node in the fleet has been rebuilt and verified healthy.

**Never flip `module_available` early.**
Specifically for `CacheModuleAvailable`: emitting `apps.cache` on a stock
`caddy:2.8` image returns `"unknown app"` and takes the node offline. The
same applies to each of the six module gates. The feature-gate pattern
is intentionally conservative (default off) so a fresh deployment is
always safe.

**Join tokens are single-use.**
`node_join_tokens.used_at` is set atomically in a transaction on redemption;
a second redeem with the same plaintext finds `used_at IS NOT NULL` and
returns 403. Tokens expire after 30 minutes regardless.

**AES-256-GCM secrets derive from `APP_SECRET`.**
All at-rest encryption uses a key derived from `APP_SECRET` via
HKDF-SHA256. Losing `APP_SECRET` means all encrypted fields (DB password
in wizard state, WG keys, DNS credentials, customer tunnel keys) are
unrecoverable. The `rotate-secret` CLI re-encrypts every field atomically
when the secret is rotated.
